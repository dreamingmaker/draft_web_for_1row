# draft_web_for_1row

Tampermonkey에서 실행하는 "1줄로 만들기" 버튼 초안입니다.

## 요구사항 정리

- 한글 웹 기안기에서 이미 작성한 텍스트를 드래그로 선택한다.
- 화면에 떠 있는 `1줄로 만들기` 버튼을 누른다.
- 선택 영역이 현재 몇 줄로 보이는지 브라우저의 렌더링 박스로 확인한다.
- 아직 2줄 이상이면 `Alt+Shift+N` 단축키를 반복 실행해서 자간을 줄인다.
- 선택 영역이 1줄이 되면 자동으로 멈춘다.
- 너무 오래 반복되거나, 단축키가 먹히지 않거나, 강제 줄바꿈 때문에 1줄이 될 수 없으면 상태 문구로 알려준다.
- 편집기가 iframe 안에 있을 수 있으므로, 최상위 버튼이 각 iframe 안의 userscript 인스턴스에 작업 요청을 보낸다.
- `진단` 버튼으로 iframe, 선택 영역, activeElement, contenteditable 후보, sandbox 여부를 확인한다.

## 파일

- `draft_web_for_1row.user.js`: Tampermonkey에 넣을 userscript

## 설치

1. Tampermonkey 대시보드에서 새 스크립트를 만든다.
2. `draft_web_for_1row.user.js` 내용을 붙여넣고 저장한다.
3. 한글 웹 기안기 페이지를 새로고침한다.
4. 텍스트를 드래그한 뒤 오른쪽 아래의 `1줄로 만들기` 버튼을 누른다.

## 진단 사용법

1. 한글 웹 기안기에서 문제되는 텍스트를 드래그한다.
2. 오른쪽 아래의 `진단` 버튼을 누른다.
3. 패널에 요약이 표시되고, 개발자도구 콘솔에 상세 JSON이 기록된다.
4. `복사` 버튼을 누르면 마지막 진단 결과를 클립보드로 복사한다.

진단 결과는 URL, element id, class name, 선택한 실제 텍스트를 그대로 기록하지 않습니다.
URL은 `host(len:n,parts:n)` 형태로, id/class/title/name은 해시로만 표시합니다.

## 테스트할 것

- 버튼 클릭 후 선택 영역이 유지되는지
- 상태 문구가 `선택 영역 찾는 중... iframe n개 확인` 다음 `선택 영역 확인` 또는 `조정 중`으로 바뀌는지
- 실제로 자간이 줄어드는지
- 1줄이 됐을 때 `완료`로 멈추는지
- 문제가 있으면 `진단` 결과에서 `userscript 응답`, `현재 DOM 선택 발견`, `저장된 선택 후보` 숫자를 확인한다.

## iframe 관련 메모

`window.frames.length`가 큰 웹 기안기는 본문 편집기가 iframe 안에 들어 있을 가능성이 큽니다.
0.2.0부터는 버튼이 최상위 페이지에 하나만 뜨고, 클릭 시 각 iframe에 작업 요청을 전달합니다.
0.3.0부터는 iframe 진단 기능이 추가되었고, `about:blank`, `about:srcdoc`, `blob:` iframe도 더 잘 잡히도록 userscript metadata를 넓혔습니다.
0.3.1부터는 대상 페이지가 JavaScript 기본 `Map`을 덮어쓴 경우에도 진단 기능이 동작하도록 내부 저장소를 배열 기반으로 바꿨습니다.
0.4.0부터는 DOM 선택 영역이 보이지 않아도 HWP 컨트롤 API 후보를 찾아 `CharShapeSpacingDecrease` 액션을 직접 실행하는 fallback을 추가했습니다.

그래도 `선택 영역을 찾지 못했습니다`가 나오면 다음 가능성이 큽니다.

- Tampermonkey가 iframe 안에서 스크립트를 실행하지 못한 경우
- 편집기 본문이 일반 DOM 선택 영역이 아니라 canvas 또는 별도 플러그인 표면인 경우
- iframe이 `sandbox` 등으로 스크립트 주입을 막는 경우
- 선택한 텍스트가 실제 웹페이지 DOM이 아니라 원격 문서 뷰어 내부 상태인 경우

## 현재 기술적 한계

브라우저 보안 정책 때문에 Tampermonkey가 만든 키보드 이벤트는 실제 키 입력과 다르게 `isTrusted: false`입니다.
한글 웹 기안기가 이 이벤트를 무시하면, 선택 영역은 찾지만 자간은 줄지 않을 수 있습니다.

그 경우에는 다음 단계로 한글 웹 기안기 내부의 자간 줄이기 버튼, 명령 API, 또는 에디터 객체를 찾아 직접 호출하는 방식으로 바꿔야 합니다.

## 문제를 좁히는 기준

- `userscript 응답`이 1개뿐이면 iframe 안에서 스크립트가 실행되지 않은 것입니다.
- `현재 DOM 선택 발견`이 0개이고 `저장된 선택 후보`도 0개이면 편집기가 DOM Selection API로 선택을 노출하지 않는 구조일 수 있습니다.
- `저장된 선택 후보`는 있는데 `현재 DOM 선택 발견`이 0개이면 버튼 클릭 과정에서 selection/focus가 사라지는 문제일 가능성이 큽니다.
- `현재 DOM 선택 발견`이 1개 이상인데 자간이 줄지 않으면 `Alt+Shift+N` 키 이벤트가 에디터에서 무시되는 문제일 가능성이 큽니다.
- `pendingDiagnostics.set is not a function` 오류가 보이면 0.3.1 이상으로 업데이트해야 합니다.
- `현재 DOM 선택 발견`이 0개이고 `HWP API 후보`가 1개 이상이면 0.4.0의 HWP API fallback이 자간 좁히기를 직접 시도합니다.

## HWP API fallback

진단 결과처럼 웹 기안기 본문이 DOM 텍스트가 아니라 HWP 전용 렌더링 표면으로 보이면 `window.getSelection()`으로 선택 영역과 줄 수를 알 수 없습니다.
이 경우 0.4.0은 포커스가 들어간 가장 안쪽 프레임에서 HWP 컨트롤 객체를 찾고, 발견되면 `CharShapeSpacingDecrease` 액션을 기본 6회 실행합니다.

이 fallback은 실제 줄 수를 확인하지 못하므로 완료 문구도 `DOM 줄 수는 확인하지 못했습니다`라고 표시합니다.
너무 많이 줄거나 부족하면 `CONFIG.apiFallbackPresses` 값을 조정해야 합니다.

## 적용 범위 조정

초기 검증을 쉽게 하려고 현재는 모든 사이트에서 실행되도록 되어 있습니다.
동작이 확인되면 `draft_web_for_1row.user.js`의 아래 줄을 실제 기안기 주소로 좁히는 것이 좋습니다.

```js
// @match        *://*/*
```
